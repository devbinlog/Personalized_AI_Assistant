## 이 프로젝트를 만든 이유

기존 AI 어시스턴트를 오래 쓰다 보면 이상한 점이 생깁니다.

내가 개발자라는 사실은 기억합니다. 하지만 내가 코드 예제를 좋아하는지, 이론 설명을 좋아하는지는 학습하지 않습니다. 오늘 짧은 답을 원했다고 해서 내일도 짧은 답을 주는 것이 아닙니다. 매번 "코드로 보여줘", "더 간단하게" 같은 지시를 반복해야 합니다.

**이 시스템은 정보가 아니라 행동을 기억합니다.**

3개의 응답 후보 중 하나를 선택하는 행동이 학습 신호가 됩니다. 코드 예제가 포함된 응답을 꾸준히 선택하면, 다음 대화에서는 코드 예제가 자동으로 늘어납니다. 말하지 않아도 됩니다.

이 차이를 만드는 것이 이 프로젝트의 시작이었습니다.

## 기존 AI와 무엇이 다른가

| | 기존 AI 어시스턴트 | 이 시스템 |
|---|---|---|
| **개인화 방식** | 사용자가 직접 지시 ("코드로 설명해줘") | 선택 행동에서 자동 학습 |
| **메모리 대상** | 이름, 직업 같은 정보 | 응답 전략, 길이, 어조, 구조 |
| **응답 생성** | 단일 응답 생성 | 3개 병렬 생성 후 평가 + 랭킹 |
| **응답 선택 근거** | 불투명 | 9차원 점수 + XAI 패널로 설명 |
| **AI의 역할** | 질문에 답하는 도구 | 사용자에게 맞춰가는 시스템 |
| **목표 달성** | 단발성 대화 | Execution Mode로 단계별 코칭 |

## 핵심 아이디어

**선택이 학습이 된다.** 사용자가 응답을 선택하는 순간, 선택된 응답의 전략 — 구조 방식, 길이, 어조, 설명 깊이 — 이 데이터가 됩니다. 이 데이터가 쌓이면 LLM이 패턴을 분석해 자연어 프로파일을 합성하고, 그 프로파일이 다음 대화의 시스템 프롬프트에 주입됩니다.

```
[학습 모드] 선택 → 학습 사이클              [일반 모드] 자동화된 AX

질문 입력                                    질문 입력
    │                                             │
    ▼                                             ▼
3개 전략으로 후보 생성                      학습된 선호도 기반
(STRUCTURED / CONCISE / PROFESSIONAL)       최적 전략 자동 선택
    │                                             │
    ▼                                             ▼
사용자가 선택                               스트리밍 응답
    │                                             │
    ▼                                             ▼
선택 전략 + 태그 → 선호도 로그              "코드로 보여줘"라고
    │                                         말하지 않아도 됨
    ▼
로그 5개 이상 누적 시
LLM이 자연어 프로파일 합성
    │
    ▼
다음 대화 시스템 프롬프트에 주입
```

```
[실행 모드] 목표 기반 코칭 — AX의 세 번째 레이어

목표 선언 ("포트폴리오 준비", "독서 습관 만들기")
    │
    ▼
Journey Planner
→ LLM이 4~8단계 실행 계획 자동 설계
→ 각 단계: 제목 + 행동 지침 + 완료 기준
    │
    ▼
각 대화마다 Progress Engine 실행
→ 현재 단계 맥락이 시스템 프롬프트에 주입
→ AI가 "몇 단계 중 몇 번째 진행 중"을 인식
    │
    ▼
여정 맥락 인식 응답
    │
    ▼
대화 후 다음 단계 자동 추천
"무엇을 해야 하는가"를 AI가 먼저 결정
```

5번 선택하면 패턴이 생깁니다. 10번 선택하면 AI가 먼저 맞춥니다.

이것은 RLHF(Reinforcement Learning from Human Feedback)의 핵심 구조를 프로덕션 수준에서 단순화한 구현입니다. Reward Model 없이, 사람의 선택 자체가 선호 신호가 됩니다.

## 세 가지 대화 모드

이 시스템은 목적에 따라 세 가지 모드로 동작합니다. 단순한 UI 옵션이 아니라, 데이터 수집 인터페이스와 AX 파이프라인 자체가 달라집니다.

### Learning Mode — 선호도 수집 인터페이스

동일한 질문에 대해 서로 다른 전략으로 생성된 3개의 응답 후보를 제시합니다. 사용자가 하나를 선택하면, 선택된 전략과 사용자가 추가하는 태그가 선호도 로그로 저장됩니다.

Learning Mode는 AX 시스템의 데이터 입력 채널입니다. 이 채널이 없으면 Normal Mode의 자동화된 개인화는 불가능합니다.

과거에 Learning Mode로 진행한 채팅방을 다시 열면 모드가 자동 복원됩니다. 사용자가 매번 모드를 다시 설정할 필요가 없습니다.

### Normal Mode — 학습 결과의 자동 적용

학습된 선호도 메모리를 바탕으로 시스템이 최적의 전략을 선택하고, 그 응답을 스트리밍합니다. 3개 후보는 내부에서 동일하게 생성 → 평가 → 랭킹되지만, 사용자에게는 최적 후보만 보입니다.

사용자가 더 이상 응답 스타일을 지시하지 않아도 되는 상태가 목표입니다.

### Execution Mode — 목표 기반 AX

"독서 습관 만들기", "포트폴리오 준비" 같은 목표를 선언하면, AI가 달성까지의 단계별 여정을 자동 설계합니다. 각 대화가 여정의 어느 단계에 해당하는지 연결되고, 대화가 끝날 때마다 다음 단계를 AI가 먼저 제안합니다.

목표가 설정되는 순간, Journey Planner가 LLM을 호출해 4~8단계의 구체적인 실행 계획을 생성합니다. 각 단계에는 제목, 행동 지침, 완료 기준이 포함됩니다. 이후 해당 목표와 연결된 모든 대화에서 Progress Engine이 현재 단계 컨텍스트를 조회하고 시스템 프롬프트에 주입합니다. AI는 "현재 어떤 목표의 몇 번째 단계를 진행 중인가"를 인식한 상태에서 응답합니다. 단발성 질문에 대한 답이 아니라, 여정 전체의 맥락 위에서 코칭합니다.

목표와 진행 상황은 대화 단위로 저장됩니다. 채팅방을 다시 열면 이전 목표와 여정 맥락이 자동 복원되고, 새 대화를 시작하면 목표가 초기화되어 다른 채팅에 영향을 주지 않습니다. 한 채팅방에서 포트폴리오 목표를 진행하는 동안, 다른 채팅방은 완전히 독립적인 일반 대화로 사용할 수 있습니다.

Execution Mode는 Learning Mode, Normal Mode와 결합됩니다. 목표를 향해 코칭하면서 동시에 사용자의 응답 선호도도 학습합니다. "무엇을 향해 가는가"와 "어떻게 응답받기를 원하는가"가 하나의 대화 안에서 함께 작동합니다.

## 핵심 AX 시스템

### 1. Task Analyzer

**왜 필요한가**

"파이썬 코드 알려줘"와 "파이썬 역사 알려줘"는 다른 방식으로 처리해야 합니다. 페르소나 선택, 웹 검색 실행 여부, 후보 생성 전략이 모두 태스크 분류 결과에서 갈립니다. 규칙 기반으로는 이 분류의 정확도를 확보하기 어렵습니다.

**어떻게 동작하는가**

경량 LLM 모델로 자연어 질문을 분류합니다. 13가지 태스크 유형, 복잡도, 도메인, 웹 검색 필요 여부, 선호 응답 전략을 단일 JSON으로 반환합니다. 호출 실패 시 `GENERAL` 폴백으로 파이프라인이 중단되지 않습니다.

```
입력: "Next.js에서 ISR 캐시 무효화하는 방법 알려줘"

출력: {
  taskType: "PROGRAMMING",
  complexity: "MODERATE",
  domain: "software",
  needsWebSearch: false,
  preferredStyle: "code-first"
}
```

**트레이드오프**

모든 메시지에 LLM 호출 비용이 발생합니다. 규칙 기반으로는 도메인 분류 정확도가 낮아, 잘못된 페르소나가 매칭되고 응답 품질이 저하됩니다. 분류 비용이 응답 품질에 직접 투자됩니다.

---

### 2. Candidate Generator

**왜 필요한가**

하나의 응답만 생성하면 그게 최선인지 알 수 없습니다. 서로 다른 전략으로 생성한 후보를 평가하고 선택해야 "더 나은 응답"이 정의됩니다.

**어떻게 동작하는가**

동일한 시스템 프롬프트에 전략 힌트를 다르게 주어 3개의 응답을 병렬 생성합니다.

| 전략 | 특성 |
|------|------|
| STRUCTURED | 헤더, 목록, 단계로 구조화 |
| CONCISE | 핵심만, 최소 분량 |
| PROFESSIONAL | 형식적, 정확한 어조 |
| ANALYTICAL | 원인-결과 분석 중심 |
| CONVERSATIONAL | 자연스러운 대화체 |

TaskType과 사용자 선호도 메모리를 기반으로 3가지 전략을 동적으로 선택합니다. 모든 호출은 `Promise.all()`로 병렬 실행됩니다.

**트레이드오프**

API 호출이 3배 발생합니다. Learning Mode에서 이 비용이 의미 있는 선호도 데이터를 만들고, Normal Mode에서는 자동 선택으로 숨겨집니다.

---

### 3. Evaluation Engine

**왜 필요한가**

3개 후보 중 어떤 것이 더 나은지 판단하는 객관적 기준이 필요합니다. "더 좋다"는 직관을 수치로 만들어야 자동 랭킹이 가능합니다.

**어떻게 동작하는가**

LLM이 9개 차원을 0–1 사이로 평가합니다.

| 차원 | 설명 |
|------|------|
| structure | 논리적 구조를 갖추었는가 |
| readability | 읽기 편하게 구성되었는가 |
| specificity | 구체적인가, 모호하지 않은가 |
| completeness | 필요한 내용이 빠지지 않았는가 |
| professionalism | 신뢰할 수 있는 어조인가 |
| formatting | 마크다운이 적절하게 사용되었는가 |
| preferenceMatch | 학습된 사용자 선호도와 맞는가 |
| taskMatch | 태스크 유형에 적합한 응답인가 |
| overall | 종합 품질 점수 |

**설계 결정: 18차원 → 9차원**

설계 초기에 18차원 루브릭을 만들었습니다. 자연스러움, 문법 정확성, 사실 정확성, 환각 위험도 등이 포함되었습니다.

문제가 있었습니다. LLM에게 "이 응답이 사실적으로 정확한가?"를 묻는 것은 같은 LLM이 생성한 응답을 같은 LLM이 검증하는 구조입니다. 자신의 출력에 대한 LLM 자기 평가는 신뢰도가 낮습니다. 측정할 수 없는 차원을 루브릭에 넣는 것은 정확도를 높이는 게 아니라 노이즈를 추가합니다. 신뢰 있게 측정 가능한 9개 차원만 남겼습니다.

---

### 4. Ranker

**왜 필요한가**

평가 점수만으로 랭킹을 결정하면 객관적 품질은 반영되지만 개인 선호는 반영되지 않습니다. "구조적으로 좋은 응답"과 "이 사용자가 좋아하는 응답"은 다를 수 있습니다.

**어떻게 동작하는가**

```
최종 점수 = 평가 점수(0~1) + 선호도 메모리 가중치(0~0.3)
```

메모리에 "STRUCTURED 전략 선호"가 기록되어 있으면 해당 전략 후보의 점수에 가중치가 추가됩니다. 품질이 동일한 두 후보가 있을 때, 사용자가 과거에 더 많이 선택한 전략이 이깁니다.

---

### 5. Preference Memory

**왜 필요한가**

선호도 로그를 그대로 프롬프트에 넣으면 토큰이 폭발합니다. 50개 로그를 그대로 넣는 것과, LLM이 50개 로그를 읽고 합성한 자연어 요약을 넣는 것은 효과와 비용이 모두 다릅니다.

**어떻게 동작하는가**

선호도 로그가 임계치(기본 5개) 이상 쌓이면, LLM이 전체 로그를 분석하고 사용자 프로파일을 합성합니다.

```
입력: 최근 선호도 로그 (선택된 전략, 태그, 태스크 유형 포함)

출력: {
  preferredTone: "casual",
  preferredLength: "concise",
  preferredStrategies: ["STRUCTURED", "CONCISE"],
  avoidedPatterns: ["장문 이론 설명"],
  rawSummary: "코드 예제 중심, 단계별 구조 선호, 이론 설명 최소화"
}
```

이 요약은 다음 대화의 시스템 프롬프트 첫 번째 섹션에 주입됩니다. 사용자가 명시적으로 지시하지 않아도 AI가 이미 알고 있는 상태를 만듭니다.

---

### 6. Prompt Builder

**왜 필요한가**

시스템 프롬프트는 텍스트가 아닙니다. 메모리 요약, 페르소나 지시사항, 태스크 컨텍스트, 웹 검색 결과, 온보딩 규칙이 순서대로 조립되는 구조입니다. 어떤 정보를 어떤 순서로 넣느냐에 따라 LLM의 응답 품질이 달라집니다.

**어떻게 동작하는가**

각 컴포넌트가 독립 블록으로 정의되고, 조건에 따라 조립됩니다.

```
시스템 프롬프트 조립 순서:
1. [BASE]        페르소나 핵심 지시사항
2. [MEMORY]      학습된 사용자 선호도 요약
3. [TASK]        현재 태스크 유형 + 복잡도
4. [FLOW]        활성 대화 플로우 + 현재 단계 지시사항 (플로우 활성 시)
5. [EXECUTION]   실행 목표 + 진행 단계 맥락 (Execution Mode 시)
6. [SEARCH]      웹 검색 결과 (needsWebSearch인 경우)
7. [ONBOARDING]  스타일 + 포맷 규칙
8. [STRATEGY]    현재 응답 전략 힌트 (후보 생성 시)
```

프롬프트 버전이 자동 저장됩니다. Prompt Lab에서 버전별 컴포넌트 구성과 토큰 수를 비교할 수 있습니다.

---

### 7. Persona System

**왜 필요한가**

같은 사람도 질문 유형에 따라 원하는 응답 스타일이 다릅니다. 기술 질문에는 코드 중심, 면접 준비에는 코칭 스타일, 리서치에는 학술적 분석이 맞습니다. 하나의 보편적 어시스턴트보다 역할에 맞는 페르소나가 응답 품질을 높입니다.

**어떻게 동작하는가**

5개의 기본 페르소나가 Task Analyzer의 결과를 기반으로 자동 매칭됩니다.

| 페르소나 | 활성화 조건 | 응답 특성 |
|----------|------------|-----------|
| Developer Mentor | PROGRAMMING, DEBUGGING, CODE_REVIEW | 코드 예제 중심, 실용적 |
| Interview Coach | CAREER, INTERVIEW | 구조화된 피드백, 코칭 어조 |
| Research Assistant | RESEARCH, ANALYSIS | 근거 기반, 다각도 분석 |
| Friendly Mentor | LEARNING, EDUCATION | 친근한 어조, 단계별 설명 |
| Professional Assistant | WRITING, PLANNING, TRANSLATION | 형식적, 정확한 |

사용자가 직접 선택하면 자동 매칭보다 우선합니다. 커스텀 페르소나 생성도 지원합니다.

### 8. XAI — 설명 가능한 AI

**왜 필요한가**

AI가 왜 이 응답을 선택했는지 알 수 없으면 시스템을 신뢰하기 어렵습니다. AX 설계에서 투명성은 신뢰의 기반입니다. 특히 개인화 시스템은 "왜 이렇게 응답했는가"가 사용자 경험의 핵심입니다.

**어떻게 동작하는가**

응답 스트리밍과 동시에 비동기로 설명 데이터를 저장합니다. 사용자는 모든 응답에서 "Why this answer?" 패널을 열어 확인할 수 있습니다.

패널에 표시되는 정보:
- 어떤 메모리가 이 응답에 영향을 미쳤는가
- 어떤 페르소나가 활성화되었는가
- 3개 후보의 9차원 점수 비교
- 왜 이 후보가 최종 선택되었는가
- 현재 적용된 선호도 가중치

### 9. Adaptive Suggestion

**왜 필요한가**

사용자가 10번 연속으로 같은 태그를 선택했다면, 시스템이 먼저 "이 설정을 기본값으로 바꾸는 게 어떨까요?"라고 제안할 수 있어야 합니다. 사용자가 설정을 수동으로 바꾸는 것이 아니라, AI가 먼저 적응을 제안하는 방향이 진정한 AX입니다.

**어떻게 동작하는가**

응답 스트리밍이 완료된 후 비동기로 실행됩니다. 최근 로그 20개를 분석하여 반복 패턴을 감지하면 제안 배너를 생성합니다.

응답 흐름과 완전히 분리되어 스트리밍 지연에 영향을 주지 않습니다. 사용자는 제안을 수락하거나 무시할 수 있습니다.

### 10. Execution Mode

**왜 필요한가**

"독서 습관 만들기"는 한 번의 대화로 해결되지 않습니다. 목표를 선언하고, 단계별로 진행하고, 각 단계에서 AI가 맥락을 유지하며 코칭하는 구조가 필요합니다.

**어떻게 동작하는가**

```
[목표 설정]
목표 입력 ──▶ /api/goals (POST)
              │
              ▼
          Journey Planner (LLM)
              │  목표 분석 → 4~8단계 여정 자동 설계
              │  각 단계: 제목 / 행동 지침 / 완료 기준 포함
              ▼
          DB: Goal 레코드 + Step[] 저장
              │
              ▼
          executionGoalId ──▶ Zustand (전역 상태)
                          ──▶ localStorage (conv_executionGoal_{conversationId})
                              ※ 대화 단위 격리 — 채팅방마다 독립 목표 유지

[대화 실행]
사용자 메시지 ──▶ /api/chat (executionGoalId 포함)
                  │
                  ▼
              Progress Engine
              ├── 현재 단계 인덱스 파악
              ├── 완료 단계 목록 조회
              └── 현재 단계 행동 지침 + 완료 기준 로드
                  │
                  ▼
              buildSystemPrompt() → [EXECUTION] 블록 추가
              ┌──────────────────────────────────────────────┐
              │ 목표: {goal.title}                             │
              │ 진행: {currentStep}/{totalSteps}단계           │
              │ 현재 단계: {step.title}                        │
              │ 행동 지침: {step.instructions}                 │
              │ 완료 기준: {step.completionCriteria}           │
              └──────────────────────────────────────────────┘
                  │
                  ▼
              LLM (여정 맥락 인식 응답 생성)
                  │
                  ▼
              SSE 스트리밍 ──▶ 사용자
                  │
                  ├── [비동기] 단계 진행 상황 업데이트 ──▶ DB
                  └── [비동기] Execution Recommendation
                              ──▶ 다음 단계 추천 배너 생성

[컨텍스트 복원]
채팅방 재진입 ──▶ localStorage 조회
                ├── 목표 있음 ──▶ setExecutionGoal(id, title) 자동 복원
                └── 목표 없음 ──▶ 일반 대화 모드
```

**Journey Planner**: 목표를 선언하면 달성까지의 단계별 여정을 설계합니다. 각 단계는 구체적인 행동 지침과 완료 기준을 포함합니다.

**Progress Engine**: 각 대화가 어떤 단계와 연결되어 있는지 추적합니다. 현재 진행 상황을 시스템 프롬프트에 주입하여 AI가 여정 맥락을 유지합니다.

**Execution Recommendation**: LangGraph StateGraph에 연결된 GPT-4o-mini가 목표·마일스톤·현재 단계 컨텍스트를 읽고, 다음 행동을 유도하는 코칭 질문을 실시간 생성합니다. /api/goals/{id}/recommend에서 FastAPI 백엔드를 우선 호출하고, 실패 시 Next.js 서비스로 폴백합니다.

**Adaptive Planning**: 진행 상황에 따라 여정 계획을 조정합니다. 예정보다 빠르거나 느린 경우 다음 단계를 재조정합니다.

**목표 컨텍스트 복원**: 실행 목표는 대화 단위로 저장됩니다. 과거에 특정 목표로 진행하던 채팅방을 다시 열면 해당 목표가 자동으로 복원됩니다. 새 채팅을 시작하면 이전 목표가 초기화되어 다른 대화에 영향을 주지 않습니다.

**설계 결정: 목표 격리의 이유**

처음에는 실행 목표를 전역 상태로만 관리했습니다. 문제는 A 채팅방에서 포트폴리오 목표를 설정하고 B 채팅방으로 이동하면 포트폴리오 목표가 B에도 주입되는 것이었습니다. 목표는 대화의 맥락입니다. 채팅방마다 독립적이어야 합니다. `conv_executionGoal_{conversationId}` 키로 localStorage에 대화 단위 격리 저장하고, 새 채팅 시작 시 자동 초기화하는 구조로 해결했습니다.

### 11. Global Learning

**왜 필요한가**

새 사용자는 선호도 데이터가 없습니다. Cold Start 문제입니다. 개인 메모리가 쌓이기 전까지 전체 사용자 패턴을 활용하면 초기 응답 품질을 높일 수 있습니다.

**어떻게 동작하는가**

Vercel Cron이 매일 새벽 3시에 실행됩니다. 모든 사용자의 선호도 로그를 집계하고, 어떤 전략이 어떤 태스크 유형에서 선호되는지를 전역 메모리로 합성합니다. 개인 메모리가 쌓이면 전역 메모리보다 우선 적용됩니다.

---

### 12. Flow Designer — 대화 플로우 설계

**왜 필요한가**

"기술 질문을 받으면 코드 예제 중심으로 답하고, 오류 메시지가 있으면 진단-해결 순서로 접근하라"는 지시를 AI에게 매번 프롬프트로 전달하는 것은 비효율적입니다. 도메인별로 AI의 대화 접근 패턴을 사전에 구조화하고, 활성화 시 자동으로 적용되는 플로우 시스템이 필요합니다.

**어떻게 동작하는가**

```
[플로우 정의]
도메인 선택 (기술 / 커리어 / 교육 / 비즈니스)
    │
    ▼
트리거 조건 설정
"코드, 오류, 버그, 개발 관련 질문"
    │
    ▼
단계별 AI 지시사항 설계
├── Step 1: 오류 진단
│     트리거 키워드: 오류, 에러, error, bug
│     지시사항: 오류 메시지 분석 → 원인 진단 → 해결책 + 코드 예시
│     검색 정책: auto
│
├── Step 2: 코드 리뷰
│     트리거 키워드: 코드, 리뷰, 리팩터
│     지시사항: 가독성/성능/보안 관점 검토 → 수정 예시 + 이유
│     검색 정책: never
│
└── Step 3: 아키텍처 논의
      트리거 키워드: 설계, 아키텍처, 패턴
      지시사항: 트레이드오프 설명 → ASCII 다이어그램 활용
      검색 정책: auto

[플로우 실행]
대화 메시지 수신
    ──▶ resolveFlow(): 트리거 키워드 매칭 → 현재 단계 결정
    ──▶ buildSystemPrompt()에 [FLOW] 블록 주입
    ──▶ LLM이 단계별 지시사항에 따라 응답
    ──▶ /api/flows/simulate: 실제 배포 전 플로우 동작 시뮬레이션
```

3가지 기본 플로우(기술 지원 / 커리어 코칭 / 학습 멘토링)가 제공됩니다. 사용자는 직접 플로우를 생성하거나 기본 플로우를 수정할 수 있습니다.

Persona System이 "어떤 어조로 말하는가"를 정의한다면, Flow Designer는 "어떤 상황에서 어떤 방식으로 접근하는가"를 정의합니다. 두 레이어가 결합하면 도메인 맞춤형 대화 패턴을 가진 AI가 만들어집니다.

### 13. Prompt Experiments — 프롬프트 A/B 실험

**왜 필요한가**

"이 프롬프트가 저 프롬프트보다 낫다"는 것을 어떻게 증명할 수 있을까요? 직관이 아닌 데이터로 결정하려면, 동일한 입력에 두 프롬프트를 실행하고 평가 기준에 따라 자동으로 승자를 결정하는 시스템이 필요합니다.

**어떻게 동작하는가**

```
실험 설계
├── 프롬프트 A: 현재 적용 중인 시스템 프롬프트
├── 프롬프트 B: 실험할 새 프롬프트
└── 테스트 입력: 비교에 사용할 질문 목록

실험 실행 ──▶ /api/prompt-experiments/{id}/run
             │
             ├── 각 테스트 입력에 대해
             │     ├── 프롬프트 A + LLM → 응답 A
             │     └── 프롬프트 B + LLM → 응답 B
             │
             ▼
         Evaluation Engine (9차원 평가)
             ├── scoreA: 응답 A 품질 점수
             └── scoreB: 응답 B 품질 점수

결과 집계
├── 입력별: A vs B 점수 비교 + 메모
├── 전체: 승패 통계 (A wins / B wins / tie)
└── 승자 프롬프트 → 이후 대화에 자동 반영
```

Prompt Lab(버전 이력 + 토큰 수 추적)과 Prompt Experiments(자동 A/B 실험)가 함께 프롬프트 엔지니어링을 주관적 판단이 아닌 실험 데이터로 관리하는 레이어를 만듭니다.

### 14. Datasets — 학습 데이터 내보내기

**왜 필요한가**

이 시스템에서 쌓이는 데이터는 AI 파인튜닝에 직접 활용할 수 있는 형태입니다. 사용자 선호도 로그는 DPO(Direct Preference Optimization) 학습 쌍으로, 평가 결과는 루브릭 기반 품질 데이터로, 대화 기록은 역할 레이블이 붙은 SFT 데이터로 변환됩니다.

**어떻게 동작하는가**

| 데이터셋 유형 | 내용 | 활용 |
|-------------|------|------|
| 선호도 데이터셋 | chosen/rejected 응답 쌍 (DPO 형식) | DPO 파인튜닝 |
| 평가 데이터셋 | 9차원 루브릭 점수 + 강점/약점 분석 | 평가 모델 학습 |
| 대화 데이터셋 | system/user/assistant 역할 레이블 대화 기록 | SFT 파인튜닝 |

내보내기 형식: **JSON / JSONL / CSV**. 레코드 수 상한과 필터 조건 지정 가능. DatasetExport 이력이 DB에 남아 어떤 데이터를 언제 내보냈는지 추적됩니다.

이 시스템은 AI를 사용하는 것에서 멈추지 않고, 사용 과정에서 생성되는 데이터를 다시 AI 학습에 활용할 수 있는 데이터 파이프라인 구조를 갖추고 있습니다.

## AX 설계 원칙

이 시스템을 설계하면서 지킨 다섯 가지 원칙입니다. 기능 결정보다 이 원칙들이 먼저였습니다.

**1. 사용자의 선택이 학습 신호다**
사용자가 직접 설정하지 않아도 된다. 선택 행동 자체가 데이터다. 인터페이스가 데이터 수집 채널이 된다.

**2. 응답 생성보다 응답 평가가 중요하다**
하나의 응답을 만드는 것보다 여러 응답 중 더 나은 것을 판단하는 시스템이 더 어렵고 더 중요하다. Evaluation Engine이 이 프로젝트의 핵심이다.

**3. 개인화는 정보 저장이 아니라 행동 변화다**
"나는 개발자야"를 기억하는 것과 코드 예제 중심으로 응답 전략이 바뀌는 것은 다르다. 전략이 바뀌어야 진짜 개인화다.

**4. 모든 AI 의사결정은 설명 가능해야 한다**
왜 이 응답이 선택되었는지 사용자가 알 수 있어야 신뢰가 생긴다. XAI는 기능이 아니라 신뢰 인프라다.

**5. AI는 사용자에게 먼저 다가와야 한다**
사용자가 AI에 맞추는 것이 아니라, AI가 사용자에게 먼저 적응을 제안해야 한다. Adaptive Suggestion이 이 방향을 구현한다.

## 기술적 결정과 트레이드오프

설계 과정에서 내린 결정들입니다. 어떤 것을 선택하고 어떤 것을 버렸는지가 시스템의 방향을 결정했습니다.

---

**18차원 평가 루브릭 → 9차원**

설계 단계에서 18차원 루브릭을 만들었습니다. 사실 정확성, 환각 위험도, 문법 정확성 등을 포함했습니다.

문제는 측정 가능성이었습니다. LLM에게 "이 응답이 사실적으로 정확한가?"를 묻는 것은 같은 LLM이 자신의 출력을 검증하는 구조입니다. 자기 참조 평가는 신뢰도가 낮습니다. 신뢰 있게 측정할 수 없는 차원을 루브릭에 넣는 것은 정확도를 높이는 것이 아니라 노이즈를 추가하는 것입니다.

9개로 줄이면서 각 차원의 정의를 더 명확하게 만들었습니다. 적은 차원, 더 정확한 평가가 더 많은 차원, 낮은 신뢰도보다 낫습니다.

---

**LangGraph를 독립 서버로 설계한 이유**

Vercel 서버리스 환경은 Python 런타임을 지원하지 않습니다. 선택지는 두 가지였습니다: LangGraph를 포기하거나, 독립 서버로 배포 가능한 구조를 유지하거나.

LangGraph의 가치는 AI 파이프라인을 StateGraph로 명시적으로 설계한다는 데 있습니다. 함수 호출 체인보다 조건부 엣지와 분기가 코드가 아닌 그래프에 드러납니다. 실행 환경보다 설계 방식이 핵심입니다.

Next.js 파이프라인이 같은 StateGraph 구조를 TypeScript로 미러링합니다. Python 백엔드는 Railway나 Render에 독립 배포 가능한 구조로 유지했습니다. 두 구현이 동일한 아키텍처 결정을 공유합니다.

---

**Mock Provider를 만든 이유**

개발 중 LLM API를 매번 호출하면 두 가지 문제가 생깁니다. 비용이 누적되고, 네트워크 지연으로 개발 사이클이 느려집니다.

Mock Provider는 규칙 기반으로 즉시 응답을 반환합니다. 파이프라인 전체가 실제 LLM 없이 동작합니다. `LLM_PROVIDER=mock` 환경 변수 하나로 전환됩니다. 코드 변경 없이 4개 Provider를 교체할 수 있는 추상화 계층이 이 설계를 가능하게 합니다.

---

**Learning Mode가 별도 모드인 이유**

Normal Mode만 있으면 사용자 피드백을 수집할 방법이 없습니다. 사용자가 응답에 불만족해도 시스템은 알지 못합니다.

Learning Mode는 선택 행동을 명시적 피드백으로 만드는 데이터 수집 인터페이스입니다. 3개의 후보를 제시하고 선택을 유도함으로써, 암묵적 선호가 명시적 로그가 됩니다. 이 데이터가 Normal Mode의 자동화를 가능하게 합니다.

---

**스트리밍 중 사이드이펙트 처리**

응답을 스트리밍하면서 DB 저장, XAI 생성, 적응형 제안 탐지를 동시에 실행해야 했습니다. 이 작업들을 응답 스트리밍 체인에 넣으면 지연이 발생합니다.

응답 스트리밍이 완료된 후, 사이드이펙트들을 비동기 체인으로 실행합니다. 사용자는 스트리밍 완료와 동시에 응답을 읽기 시작할 수 있고, 백그라운드에서 XAI와 제안이 저장됩니다.

---

**프롬프트 버전 관리를 구현한 이유**

"이 프롬프트가 지난 버전보다 나은가?"를 직관으로 판단하는 대신, 같은 입력에 두 버전을 실행하고 9차원 평가 점수로 승자를 결정합니다. 승자 프롬프트는 이후 채팅에 자동 반영됩니다.

이것이 Prompt Lab의 A/B 실험 기능입니다. 프롬프트 엔지니어링을 주관적 판단이 아닌 데이터 기반 결정으로 만듭니다.

---

**실행 목표를 전역 상태가 아닌 대화 단위로 격리한 이유**

처음에는 실행 목표를 Zustand 전역 상태로만 관리했습니다. 사용자가 A 채팅방에서 "포트폴리오 준비" 목표를 설정하고 B 채팅방으로 이동하면, 포트폴리오 맥락이 B 채팅방에도 주입되는 문제가 생겼습니다.

목표는 대화의 맥락입니다. 채팅방 간에 공유되어서는 안 됩니다. `conv_executionGoal_{conversationId}` 키로 localStorage에 대화 단위로 격리 저장하고, 새 채팅 시작 시 자동 초기화하는 구조로 해결했습니다. Zustand는 현재 활성 대화의 목표만 보유하고, 영속성은 localStorage가 담당하는 역할 분리입니다.

## 기술적 도전

**userId 계층 해결**

같은 사용자가 로그인 상태와 비로그인 상태를 오갈 때 선호도 데이터가 분리됩니다. NextAuth 세션, 쿠키 기반 세션 ID, anonymous 폴백을 3단계로 처리하는 `resolveUserContext()`를 구현했습니다. 어떤 상태에서도 동일한 사용자로 인식됩니다.

**LLM 평가의 자기 참조 편향**

LLM에게 LLM 응답을 평가하게 하면 자신의 스타일을 더 높게 평가할 수 있습니다. 사용자의 실제 선택 데이터를 평가 결과에 가중치로 반영하여 편향을 보정합니다. 주관적 선호가 객관적 평가를 조정합니다.

**Prisma 배열 필드 JSON 직렬화**

`preferredStrategies`, `avoidedPatterns` 같은 배열 필드를 `String @default("[]")` 컬럼에 JSON으로 직렬화해 저장합니다. PostgreSQL 네이티브 배열 타입 대신 이 패턴을 선택한 이유는 Prisma의 타입 시스템과의 일관성 때문입니다. `JSON.stringify()`로 쓰고 `JSON.parse()`로 읽는 패턴이 누락되면, 런타임에 JSON 문자열이 배열처럼 순회되는 버그가 생깁니다. 24개 DB 테이블에서 이 패턴이 반복되므로 레이어 별 유틸리티 함수로 추상화했습니다.

**Zod v3/v4 호환성**

Vercel AI SDK의 `generateObject`는 Zod v3 스키마만 지원합니다. v4 마이그레이션 시 런타임 오류가 발생합니다. 프로젝트 전체를 Zod v3로 고정하고, 패키지 의존성 설정으로 버전 혼용을 방지합니다.

## 데이터 흐름

### Normal Mode

```
사용자 ──▶ /api/chat ──▶ DB (메모리 조회)
                    ──▶ LLM (태스크 분석)
                    ──▶ LLM x3 (후보 병렬 생성)
                    ──▶ LLM (9차원 평가)
                    ──▶ rankCandidates()
                    ──▶ SSE 스트리밍 ──▶ 사용자
                    ──▶ [비동기] LLM (XAI 생성) ──▶ DB
                    ──▶ [비동기] detectSuggestions() ──▶ DB
```

### Learning Mode

```
사용자 ──▶ /api/chat ──▶ [동일 파이프라인]
                    ──▶ 후보 3개 반환 ──▶ 사용자 선택 UI

사용자 선택 ──▶ /api/preferences
              ──▶ DB (선호도 로그 저장)
              ──▶ 로그 5개 이상?
                    ├── Yes ──▶ LLM (메모리 합성) ──▶ DB
                    └── No  ──▶ 완료
```

### Execution Mode

```
1단계: 목표 설정

사용자 ──▶ /api/goals (POST)
            │
            ▼
        Journey Planner (LLM)
            │  목표 분석 + 4~8단계 여정 설계
            ▼
        DB: Goal + Step[] 저장
            │
            ▼
        executionGoalId ──▶ Zustand (전역 상태)
                        ──▶ localStorage (conv_executionGoal_{conversationId})
                            ※ 대화 단위 격리 저장

2단계: 목표 컨텍스트 대화

사용자 ──▶ /api/chat (executionGoalId 포함)
            ──▶ DB (Goal + 현재 Step 조회)
            ──▶ Progress Engine
                  │  현재 단계 컨텍스트 로드
                  ▼
            buildSystemPrompt()
            [EXECUTION 블록 주입]
            ├── 현재 목표: {goal.title}
            ├── 진행 단계: {n}/{total}
            └── 행동 지침 + 완료 기준
                  │
                  ▼
            LLM x3 (목표 맥락 인식 후보 생성)
                  │
                  ▼
            rankCandidates() ──▶ SSE 스트리밍 ──▶ 사용자
                  │
                  ├── [비동기] 단계 진행 상황 ──▶ DB 업데이트
                  └── [비동기] Execution Recommendation
                              ──▶ 다음 단계 배너 ──▶ UI

3단계: 컨텍스트 복원

채팅방 재진입
    ──▶ localStorage (conv_executionGoal_{conversationId})
          ├── 저장된 목표 있음
          │     ──▶ setExecutionGoal(id, title) 자동 복원
          │     ──▶ 이전 여정 맥락 + 진행 단계 복원
          └── 저장된 목표 없음
                ──▶ 일반 대화 모드 (목표 없이 진행)
```

## 배운 것

**AX는 기능이 아니라 설계다**

처음에는 LLM API를 호출해 응답을 반환하는 것이 전부라고 생각했습니다. 실제로 만들면서 응답 품질보다 사용자가 응답을 어떻게 경험하는지가 더 중요하다는 것을 배웠습니다. 스트리밍 속도, 후보 선택 인터페이스, XAI 투명성이 모두 AX의 일부입니다. AI 챗봇을 만드는 것과 AI 경험 시스템을 설계하는 것은 다릅니다.

**프롬프트 엔지니어링은 소프트웨어 엔지니어링이다**

프롬프트는 텍스트가 아닙니다. 메모리, 페르소나, 태스크 컨텍스트, 검색 결과, 온보딩 규칙이 순서대로 조립되는 구조입니다. 어떤 정보를 어떤 순서로 넣느냐에 따라 응답 품질이 달라집니다. 버전 관리, A/B 실험, 컴포넌트 추상화가 필요한 이유가 여기 있습니다.

**평가를 설계하는 것이 가장 어렵다**

"좋은 응답"을 정의하는 것 자체가 문제입니다. 객관적 품질 기준과 주관적 선호가 충돌합니다. 측정 가능한 기준과 측정 불가능한 기준을 구분해야 합니다. 18차원에서 9차원으로 줄인 과정이 이 문제를 직접 경험한 결과입니다.

**비동기 사이드이펙트 설계가 UX를 결정한다**

스트리밍 응답과 DB 저장, XAI 생성을 같은 체인에 넣으면 사용자가 느끼는 응답 속도가 달라집니다. 어떤 작업을 응답 경로에 넣고 어떤 작업을 비동기로 분리할지가 UX 설계 결정입니다.

**목표 기반 AI 설계는 단발성 QA 설계와 근본적으로 다르다**

Learning Mode와 Normal Mode를 설계할 때는 "이 질문에 어떻게 답하는가"가 중심이었습니다. Execution Mode를 설계하면서 "사용자가 무엇을 향해 가는가"를 먼저 묻는 것이 다른 설계 문제라는 것을 배웠습니다. 여정 맥락을 시스템 프롬프트에 주입하는 것이 기술적으로는 단순해 보이지만, 그 맥락이 없으면 AI는 매 대화를 독립적 이벤트로 처리합니다. 대화와 대화를 이어주는 것이 코칭과 QA의 차이입니다.

**좋은 AI 시스템은 더 많은 기능이 아니라, 사용자의 행동을 더 나은 경험으로 연결하는 구조다.**

---

## 마치며

이 프로젝트의 목표는 "AI 챗봇을 만드는 것"이 아니었습니다.

사용자의 선택이 어떻게 학습 신호가 되는지, 그 신호가 어떻게 응답 전략을 바꾸는지, 바뀐 전략이 어떻게 사용자 경험을 개선하는지 — 이 흐름 전체를 직접 설계하고 구현하는 것이 목표였습니다.

응답을 생성하는 것보다 응답을 평가하는 것이 더 어렵습니다. 정보를 기억하는 것보다 행동을 학습하는 것이 더 의미 있습니다. AI가 답하는 것보다 AI가 먼저 맞춰오는 것이 더 나은 경험입니다.

이 세 가지 방향이 이 시스템의 모든 결정의 출발점이었습니다.
